# 로봇팔 컨베이어 게임 — 설계 문서 (v1)

## 목적

포트폴리오용 프로젝트. 저장소 소유자는 백엔드 개발자이며, 이 프로젝트의 1차 목표는 **서버/프로토콜 설계 역량**을 보여주는 것이다. 게임성과 "로봇 제어를 이해한다"는 인상은 2차 목표로, 클라이언트는 서버가 계산한 상태를 동기화·표현하는 데 집중하고 판단 로직을 갖지 않는다.

- 로봇축구 레포와는 완전히 별개 프로젝트 (폴더 통합도 하지 않음).
- "작지만 완성도" 원칙 유지. 지금은 이 저장소에서 문서화 + 로컬 실험만 진행하고, 공개 시점에 별도 레포로 정리한다.
- 실시간 멀티플레이/리더보드가 목적이 아니라, 서버 권위 아키텍처 자체를 보여주기 위해 서버를 포함한다.

## 컨셉

공장 컨베이어 벨트 라인을 배경으로, 4족 보행 로봇(어깨에 장착된 긴 팔 포함)이 라인을 돌아다니며 작업을 돕는다. 플레이어는 공장 관리자이자, 필요하면 로봇 하나를 직접 조작할 수 있는 오퍼레이터다.

### 플레이어 상호작용

- **관리자 레이어** (경영 게임의 최소 형태): 컨베이어 on/off, 로봇 대수 증감. 목표는 생산량/가동률 유지.
- **직접 제어**: 로봇 하나를 "클레임"하여 팔 동작(집기/놓기 등)을 수동으로 트리거. 이동은 항상 자동(경로탐색)이며 직접 조종 대상이 아니다.

### 로봇 모델

- 4족 보행 몸체 + 어깨에 장착된 긴 팔(2~3관절).
- 몸체 자세(웅크리기/서기)와 팔 IK 타겟을 조합해 다양한 포즈 표현 (예: 낮은 컨베이어는 웅크려서, 높은 곳은 서서 팔을 뻗어 작업).
- 보행(gait)은 프로시저럴 방식: 다리 4개에 위상차를 둔 사인파/상태머신으로 발 위치 계산. 사전 제작 애니메이션 대비 "제어"하는 인상이 강함.
- 팔 IK는 2-본 해석적 솔버(어깨-팔꿈치-손목)로 시작. 범용 다관절 솔버(FABRIK 등)는 v2.

### 경로탐색

- 공장 바닥을 그리드로 표현, A* 기반 탐색.
- 장애물 회피 대상: 다른 로봇, 정지된 컨베이어 구간 등.
- v1의 스타 기능 — 로봇이 실제로 길을 찾고 서로를 피해 다니는 모습이 핵심 시각적 어필 포인트.

## 아키텍처

```
Rust 서버 (권위)  ──WebSocket──  TS/Vite/Canvas 클라이언트
  - 컨베이어 상태/속도
  - 로봇 상태머신 (위치, 자세, 작업, 배터리)
  - 경로탐색 (A* on grid)
  - 생산/처리량 계산 (경영 로직)
  - 틱 기반 결정적 시뮬레이션
                                    - 아이소메트릭 투영 렌더링
                                    - 입력 전송 (컨베이어 on/off, 로봇 수, 로봇 클레임+팔 동작)
                                    - 서버 상태 보간(interpolation) 후 표시
```

로봇축구와 동일한 "서버 권위 + 결정적 코어" 패턴을 재사용한다. 클라이언트는 동기화·보간·렌더링만 담당하고 게임 판단 로직을 갖지 않는다.

## 프로토콜 & 백엔드 설계 (핵심 어필 포인트)

### WebSocket 프로토콜

- **클라이언트 → 서버**: 커맨드 메시지 (`SelectRobot`, `ReleaseRobot`, `ToggleConveyor`, `SetRobotCount`, `TriggerArmAction`). 서버가 모든 커맨드를 검증한다 (예: 존재하지 않는 로봇 선택 방지, 이미 클레임된 로봇 재클레임 방지).
- **서버 → 클라이언트**: 틱 단위 상태 브로드캐스트. 델타 동기화로 설계 — 매 틱 전체 상태 대신 변경된 필드만 전송.
- 메시지에 버전 필드(`v: 1`)를 포함해 프로토콜 확장성을 명시적으로 설계. 시작은 JSON(가독성/디버깅), 문서에 바이너리(bincode 등) 전환 트레이드오프를 남겨둔다.
- **재접속**: 세션 토큰(UUID) 발급. 연결이 끊겨도 유예시간(예: 30초) 동안 서버가 세션 상태를 유지하고, 유예시간 내 재접속 시 풀 스냅샷 재전송 후 델타 스트림 재개.
- **백프레셔**: 클라이언트가 느리게 소비할 경우 무한 큐잉 대신 최신 상태로 coalesce하는 바운디드 채널 정책.

### 동시성 모델

- 틱 기반 고정 타임스텝 시뮬레이션.
- 매 틱마다 로봇 N대의 경로탐색/상태 갱신을 병렬 계산(Rayon 등). 각 로봇은 다른 로봇의 위치를 read-only로만 참조(장애물 회피)하고 자신의 상태만 갱신 — 데이터 경쟁 없는 동시성 모델.
- 전역 커맨드(컨베이어 on/off, 로봇 수 조절)는 단일 틱 루프가 순차 처리하므로 레이스 없음.

### 세션 / 동시접속 관리

- 단일 공유 "공장 인스턴스"에 여러 관측자가 동시 접속 가능.
- 로봇 직접 제어는 클레임 방식: 한 세션이 로봇을 잡으면 타임아웃 전까지 다른 세션은 해당 로봇을 조작할 수 없음(낙관적 락 + 타임아웃 자동 해제).

### 영속화 & API

- 생산/경영 지표(생산량, 가동률, 로봇별 작업량)를 주기적으로 SQLite에 적재 (필요 시 Postgres로 전환 가능하도록 설계).
- REST API로 실시간(WS)과 조회/설정(REST) 책임을 분리:
  - `GET /api/stats/history`
  - `GET /api/config`, `POST /api/config`

### 관찰가능성

- `tracing` 기반 구조화 로깅 (틱 번호, 로봇 id, 이벤트 타입 필드 포함).
- `/metrics` 엔드포인트 (Prometheus 포맷): 틱레이트, 접속 클라이언트 수, 로봇 수, 틱당 평균 경로탐색 시간, DB 쓰기 지연.

### 결정적 코어 + 테스트 전략

- 경로탐색·틱 갱신·IK 계산·생산량 계산을 순수 함수로 분리: `(state, commands) -> new_state`, I/O 없음.
- 유닛테스트: 그리드+장애물+커맨드 조합에서 경로가 벽을 넘지 않는지, 생산량 공식이 기대값과 일치하는지 검증.
- 프로퍼티 기반 테스트(proptest 등)로 경로탐색이 항상 종료되고 충돌하지 않는지 검증.
- 통합테스트: 서버를 띄우고 가짜 WS 클라이언트로 커맨드 시퀀스를 보내 델타 상태가 기대와 일치하는지 확인.

## 렌더링

- 2D 그리드 좌표를 고정 각도의 아이소메트릭 투영으로 변환해 그린다.
- z-order 정렬은 `x + y` 기준.
- 상태/센서 시각화 HUD: 선택(클레임)한 로봇의 배터리, 현재 작업, 탐색 경로(디버그 라인) 표시.

## v1 범위

| 영역 | v1 포함 |
|---|---|
| 로봇 | 4족 보행(프로시저럴 gait) + 긴 팔(2-본 IK), 기본 자세 |
| 경로탐색 | A* + 장애물 회피 (스타 기능) |
| 경영 | 컨베이어 on/off, 로봇 수 증감, 생산량 목표 (최소 레이어) |
| 백엔드 | Rust 서버, WS 델타 동기화+버전필드, 틱 기반 병렬 시뮬레이션, REST(설정/통계), SQLite 영속화, 재접속 유예, `/metrics`, 로봇 클레임 락, 결정적 코어+테스트 |
| 렌더링 | 아이소메트릭 투영, 상태/센서 시각화 HUD |

## v2 백로그 (범위 밖)

- 몸체+팔 조합 포즈 고도화 (범용 다관절 IK, 예: FABRIK)
- 경제 시스템 확장 (자원/비용/투자)
- Postgres 전환, Grafana 대시보드 연동
- 서버 웜 리스타트 실제 구현 (v1은 설계만)

## 스코프 가드레일

- 로봇축구 레포엔 절대 얹지 않는다 (모드 통합 X, 폴더 분리도 비추천).
- 공개할 차례가 오기 전까지는 새 레포를 만들지 않고 이 저장소에서 로컬로만 진행한다.
