# 로봇팔 컨베이어 게임 — 설계 문서 (v1)

## 목적

포트폴리오용 프로젝트. 저장소 소유자는 백엔드 개발자이며, 이 프로젝트의 1차 목표는 **서버/프로토콜 설계 역량**을 보여주는 것이다. 게임성과 "로봇 제어를 이해한다"는 인상은 2차 목표로, 클라이언트는 서버가 계산한 상태를 동기화·표현하는 데 집중하고 판단 로직을 갖지 않는다.

- 로봇축구 레포와는 완전히 별개 프로젝트 (폴더 통합도 하지 않음).
- "작지만 완성도" 원칙 유지. 지금은 이 저장소에서 문서화 + 로컬 실험만 진행하고, 공개 시점에 별도 레포로 정리한다.
- 실시간 멀티플레이/리더보드가 목적이 아니라, 서버 권위 아키텍처 자체를 보여주기 위해 서버를 포함한다.
- **로봇축구와의 차별점** (두 프로젝트를 함께 공개하게 될 경우 대비): "서버 권위 + 결정적 코어" 골격은 재사용하지만, 이 프로젝트가 새로 증명하는 것은 델타 동기화 프로토콜, 세션/재접속 처리, 관측성(로깅/메트릭), 로봇 기구학(보행+IK+경로탐색)이다. 둘 다 공개하는 시점이 오면 README에 신규 증명 포인트를 명시해 "같은 뼈대 재탕"으로 읽히지 않게 한다. 로봇축구를 공개하지 않기로 결정하면 이 항목은 무의미해지므로 그때 삭제한다.

## 컨셉

공장 컨베이어 벨트 라인을 배경으로, 4족 보행 로봇(어깨에 장착된 긴 팔 포함)이 라인을 돌아다니며 작업을 돕는다. 플레이어는 공장 관리자이자, 필요하면 로봇 하나를 직접 조작할 수 있는 오퍼레이터다.

### 플레이어 상호작용

- **관리자 레이어** (경영 게임의 최소 형태): 컨베이어 on/off, 로봇 대수 증감. 목표는 생산량/가동률 유지.
- **직접 제어**: 로봇 하나를 선택해 팔 동작(집기/놓기 등)을 수동으로 트리거. 이동은 항상 자동(경로탐색)이며 직접 조종 대상이 아니다. (v1은 단일 오퍼레이터 세션이므로 "선택"은 락이 필요 없는 단순 상태다 — 세션 관리 절 참고.)

### 로봇 모델

몸체와 팔을 하나의 기구학 체인으로 설계한다: 몸체 자세(위치+높이)가 루트 트랜스폼이고, 어깨 관절은 그 자식이며, 팔 IK 타겟은 항상 **몸체 로컬 좌표계**로 표현한다. 몸체가 웅크리거나 설 때마다 어깨 위치가 갱신되고 IK가 그 위치를 기준으로 다시 풀리므로, 팔이 몸체 기준으로 붕 뜨거나 따로 노는 문제를 원천적으로 막는다.

- 4족 보행 몸체 + 어깨에 장착된 긴 팔(2~3관절).
- 몸체 자세(웅크리기/서기)와 팔 IK 타겟(몸체 로컬 좌표)을 조합해 다양한 포즈 표현 (예: 낮은 컨베이어는 웅크려서, 높은 곳은 서서 팔을 뻗어 작업).
- 보행(gait)은 프로시저럴 방식: 다리마다 스탠스(지면 지지)/스윙(이동) 상태를 구분하는 상태머신 + 위상차(트롯 기준 대각선 다리 쌍이 ½ 주기 차이)로 발 위치 계산. 회전 시에는 다리별 보폭을 좌우 비대칭으로 조정한다. 단순 사인파만으로는 발이 미끄러지듯 보이므로 스탠스/스윙 구분이 핵심.
- 팔 IK는 2-본 해석적 솔버(어깨-팔꿈치-손목)로 시작하며, 타겟이 도달 불가능한 거리일 때 팔 길이 한계로 클램프하는 처리를 포함한다. 범용 다관절 솔버(FABRIK 등)와 정교한 pole-vector 제어는 v2.

### 경로탐색

- 공장 바닥을 그리드로 표현, A* 기반 탐색.
- 장애물 회피 대상: 다른 로봇, 정지된 컨베이어 구간 등.
- 정적 A*만으로는 여러 로봇이 서로의 계획을 모른 채 재계획을 반복해 진동(서로 번갈아 길을 막았다 비켰다)하는 문제가 생긴다. 이를 막기 위해:
  - 매 틱 전체가 아니라 일정 주기(예: N틱마다)로만 재계획하고, 그 사이는 계획된 경로를 그대로 따라간다.
  - 다음 한 칸만 예약(reservation)하는 가벼운 방식으로 정면 충돌을 막는다 (풀 time-expanded 탐색 같은 고급 기법은 쓰지 않음 — 포트폴리오 스코프에 과함).
- 로봇이 실제로 길을 찾고 서로를 피해 다니는 모습은 백엔드 아키텍처(결정적 코어, 병렬 틱 처리)를 시각적으로 증명하는 데모 역할을 한다. 이 문서와 발표에서는 이걸 "게임의 재미 포인트"가 아니라 "백엔드가 실제로 계산한 결과를 보여주는 창"으로 프레이밍한다.

## 아키텍처

```
Rust 서버 (권위)  ──WebSocket──  TS/Vite/Canvas 클라이언트
  - 컨베이어 상태/속도
  - 로봇 상태머신 (위치, 자세, 작업, 배터리)
  - 경로탐색 (A* on grid)
  - 생산/처리량 계산 (경영 로직)
  - 틱 기반 결정적 시뮬레이션
                                    - 아이소메트릭 투영 렌더링
                                    - 입력 전송 (컨베이어 on/off, 로봇 수, 로봇 선택+팔 동작)
                                    - 서버 상태 보간/외삽 후 표시
```

로봇축구와 동일한 "서버 권위 + 결정적 코어" 패턴을 재사용한다. 클라이언트는 동기화·보간·렌더링만 담당하고 게임 판단 로직을 갖지 않는다.

## 프로토콜 & 백엔드 설계 (핵심 어필 포인트)

### WebSocket 프로토콜

v1은 **단일 오퍼레이터 세션**을 전제로 한다 — 한 번에 하나의 클라이언트만 조작 권한을 갖는다(관전용 다중 접속은 v2 백로그). 이렇게 스코프를 좁히면 클라이언트별 델타 베이스라인 동기화 같은 다중 클라이언트 정합성 문제를 v1에서 아예 제거할 수 있다.

- **클라이언트 → 서버**: 커맨드 메시지 (`SelectRobot`, `ReleaseRobot`, `ToggleConveyor`, `SetRobotCount`, `TriggerArmAction`). 서버가 모든 커맨드를 검증한다 (예: 존재하지 않는 로봇 선택 방지).
- **서버 → 클라이언트**: 틱 단위 상태 브로드캐스트. 델타 동기화로 설계 — 매 틱 전체 상태 대신 직전에 **이 클라이언트에게 성공적으로 전송한 상태** 대비 변경된 필드만 전송한다 (전역 델타가 아니라 클라이언트별 기준선). 틱 레이트는 20Hz(50ms)로 목표한다.
- 메시지에 버전 필드(`v: 1`)를 포함해 프로토콜 확장성을 명시적으로 설계. 시작은 JSON(가독성/디버깅), 문서에 바이너리(bincode 등) 전환 트레이드오프를 남겨둔다 — v1에서 실제로 구현하지는 않고 트레이드오프만 문서화한다.
- **재접속**: 세션 토큰(UUID) 발급. 연결이 끊겨도 유예시간(예: 30초) 동안 서버가 세션 상태(델타 기준선 포함)를 유지하고, 유예시간 내 재접속 시 풀 스냅샷 재전송 후 델타 스트림 재개. 유예시간 초과 시 다음 접속자가 새 오퍼레이터 세션을 시작할 수 있다.
- **백프레셔**: 클라이언트가 느리게 소비할 경우, 밀린 델타들을 순서대로 쌓지 않고 "마지막으로 보낸 기준선 대비 현재 상태"로 다시 계산한 단일 델타로 합쳐서(coalesce) 전송한다 — 클라이언트별 기준선 방식이라 이 축약이 정합성을 깨지 않는다.

### 동시성 모델

- 틱 기반 고정 타임스텝 시뮬레이션 (20Hz 목표).
- 매 틱마다 로봇 N대의 경로탐색/상태 갱신을 병렬 계산(Rayon 등). 각 로봇은 **이전 틱 종료 시점에 얼어붙은(freeze) 스냅샷**만 read-only로 참조(장애물 회피)하고, 이번 틱에서 계획한 다음 위치는 별도 버퍼에 쓴다 (더블 버퍼링) — 이번 틱 갱신 중인 다른 로봇의 "계획 중" 상태를 절대 참조하지 않으므로 데이터 경쟁이 없다.
- **동시 이동 충돌 처리**: 두 로봇이 같은 틱에 같은 칸으로 이동을 계획하면, 로봇 ID가 낮은 쪽이 이동하고 나머지는 이번 틱 이동을 취소하고 다음 틱에 재계획한다 (결정적 타이브레이크).
- 생산량 등 여러 로봇의 결과를 합산하는 집계는 항상 로봇 ID 정렬 순서로 reduce해 부동소수점 합산 순서에 의한 비결정성을 없앤다.
- 전역 커맨드(컨베이어 on/off, 로봇 수 조절)는 단일 틱 루프가 순차 처리하므로 레이스 없음.
- 한 로봇의 경로탐색/IK 계산이 패닉하면 해당 로봇만 이번 틱 갱신을 건너뛰고 에러를 로깅한다 — 틱 전체가 죽지 않도록 각 로봇 갱신을 `catch_unwind`로 격리한다.

### 세션 관리

- v1은 단일 오퍼레이터 세션 전제(위 프로토콜 절 참고) — 동시 접속 자체를 다루지 않으므로 세션 간 락 경쟁이 없다.
- "로봇 선택(select)"은 이 단일 세션 내에서 현재 조작 대상이 어느 로봇인지 나타내는 서버 측 상태일 뿐이며, 클레임/락 메커니즘은 필요 없다. 여러 관측자가 동시에 같은 로봇을 다투는 문제는 v2에서 다중 접속을 도입할 때 다시 설계한다(그때는 뮤텍스+타임아웃 기반 락으로 명명을 정확히 한다 — "낙관적 락"이 아니라 "타임아웃부 배타 락").

### 영속화 & API

- 생산/경영 지표(생산량, 가동률, 로봇별 작업량)를 주기적으로 SQLite에 적재.
- 주기적으로(예: 100틱마다) 로봇 위치/자세/작업 상태를 포함한 **전체 시뮬레이션 스냅샷**도 함께 적재한다 — v1에서 웜 리스타트를 구현하지는 않지만, 이 스냅샷 스키마가 있어야 v2에서 실제로 구현할 수 있다. 지금은 집계 통계만 있고 스냅샷이 없으면 v2 웜 리스타트는 처음부터 다시 설계해야 하므로, 스키마만 지금 만들어 둔다.
- REST API로 실시간(WS)과 조회/설정(REST) 책임을 분리:
  - `GET /api/stats/history`
  - `GET /api/config`, `POST /api/config`
- Postgres 전환은 v1에서 구현하지 않고, "스키마가 SQLite 종속적이지 않게 짠다" 정도의 설계 원칙만 지킨다(v2 백로그).

### 관찰가능성

- `tracing` 기반 구조화 로깅 (틱 번호, 로봇 id, 이벤트 타입 필드 포함).
- `/metrics` 엔드포인트 (Prometheus 포맷): 틱레이트, 로봇 수, 틱당 평균 경로탐색 시간, DB 쓰기 지연. Grafana 대시보드 연동은 만들지 않는다(v2) — 엔드포인트 존재와 지표 설계 자체가 요점이지, 시각화 인프라 구축은 포트폴리오 스코프를 벗어난다.

### 성능 목표 (수치 기반 검증)

"Rayon으로 병렬화했다"가 근거 없는 버즈워드로 읽히지 않도록, v1 완료 기준에 실측 목표를 둔다:

- 로봇 20~50대 동시 시뮬레이션 유지.
- 틱 레이트 20Hz(50ms) 안정적으로 유지 (지터 허용 범위 명시).
- 틱 처리 시간 p99 < 10ms (측정은 `/metrics`로 노출한 틱당 처리시간 지표 사용).
- 위 수치를 README에 실측 그래프/로그와 함께 기록해, "설계만 하고 검증 안 함"이라는 인상을 피한다.

### 결정적 코어 + 테스트 전략

- 경로탐색·틱 갱신·IK 계산·생산량 계산을 순수 함수로 분리: `(state, commands) -> new_state`, I/O 없음.
- 유닛테스트: 그리드+장애물+커맨드 조합에서 경로가 벽을 넘지 않는지, 생산량 공식이 기대값과 일치하는지 검증.
- 프로퍼티 기반 테스트(proptest 등)로 경로탐색이 항상 종료되고 충돌하지 않는지 검증.
- 통합테스트: 서버를 띄우고 가짜 WS 클라이언트로 커맨드 시퀀스를 보내 델타 상태가 기대와 일치하는지 확인.
- **장애 주입 테스트**: 재접속이 헤드라인 기능이므로, 통합테스트에서 실제로 연결을 끊었다 재연결시켜 풀 스냅샷+델타 재개가 정확한지 검증한다. 로봇 갱신 중 패닉을 강제로 발생시켜 해당 틱이 죽지 않고 다른 로봇은 정상 갱신되는지도 검증한다.
- CI에서 위 테스트(유닛/프로퍼티/통합/장애주입)를 모두 실행한다.

### 알려진 v1 범위 밖 항목 (의도적 생략)

- 어드민 커맨드(컨베이어 on/off, 로봇 수 조절)에 대한 인증/레이트리밋 없음 — 단일 오퍼레이터 세션 전제이므로 v1에서는 불필요하다고 판단. 다중 접속을 도입하는 v2에서 재검토.

## 렌더링

- 2D 그리드 좌표를 고정 각도의 아이소메트릭 투영으로 변환해 그린다.
- z-order 정렬은 기본적으로 `x + y` 기준이되, 팔이 인접 타일 위로 뻗을 수 있으므로 로봇 오브젝트의 정렬 키는 몸체 칸이 아니라 (몸체+팔이 차지하는) 바운딩 박스의 가장 먼 안쪽 모서리 기준으로 계산한다. 이렇게 해야 팔이 인접 타일의 다른 오브젝트를 넘어갈 때 z-fighting이 생기지 않는다.
- 서버 상태 보간(interpolation): 서버 틱(20Hz)과 클라이언트 렌더 프레임레이트가 다르므로, 클라이언트는 최근 2개 틱 상태 사이를 렌더 프레임마다 선형 보간해 부드럽게 표시한다. 네트워크 지연으로 다음 틱이 늦게 도착하면 마지막 알려진 속도로 짧게 외삽(extrapolation)하고, 실제 틱이 도착하면 위치를 스냅 보정한다.
- 상태/센서 시각화 HUD: 선택한 로봇의 배터리, 현재 작업, 탐색 경로(디버그 라인) 표시.

> **각주(Plan 4, 2026-07-18 완료)**: `facing`/`arm_pose`/`path`가 `protocol.rs`의 `RobotView`에 실제로 노출되고 클라이언트가 이를 소비하기 시작한 시점은 Plan 4(Task 2 `188b23c`, Task 6/9의 `render/projection.ts`+`render/canvas.ts`)다. 그 전까지 `ik.rs`/`posture.rs`(2-본 IK, task→자세 매핑)는 Plan 1부터 존재했지만 자기 자신의 유닛테스트에서만 호출되는 죽은 코드였다 — 실제 런타임에서 최초로 불린 것이 이 시점이다. `facing`은 Plan 4 Task 1(`cc8cfa2d`)에서 `sim_core::sim::Robot`에 새로 추가된 필드다.

## 발표/데모 전략

리뷰어(채용 패널)는 Rust 소스를 읽지 않는다는 전제로 계획한다.

- **README 내러티브**: 무엇을 왜 만들었는지(백엔드 역량 증명이 1차 목표라는 이 문서의 "목적" 절 내용)를 첫 문단에 명시. 델타 동기화/백프레셔/재접속 같은 트레이드오프를 짧은 절로 설명(코드가 아니라 글로 증명).
- **데모 GIF/짧은 영상**: 로봇이 경로탐색+보행+IK로 작업하는 모습, 컨베이어 on/off, 재접속 복구 장면을 30초 내외로.
- **배포**: Docker 이미지로 패키징하고, 작은 호스트(Fly.io/Railway 등)에 올려 라이브 URL 하나로 접근 가능하게 한다. 로컬 클론 없이 브라우저에서 바로 체험 가능해야 한다.
- **퀵스타트**: README에 "이 URL을 열면 바로 볼 수 있음" + 로컬 실행 시 "`docker compose up` 한 줄"을 명시.
- **성능 목표 절과 연결**: 위 "성능 목표"에서 실측한 수치(로봇 수, 틱레이트, p99)를 README에도 그대로 인용.

> **각주(Plan 5, 2026-07-18 완료)**: 단일 컨테이너 Docker 패키징(서버가 `tower-http::ServeDir`로 클라이언트 정적 파일까지 서빙, `GAMEROBOTFACTORY_BIND_ADDR`로 바인드 주소 설정 가능), `fly.toml`(Fly.io 배포 설정), CI `docker build` 스모크 잡, Playwright 기반 데모 영상 자동 녹화(`client/scripts/record-demo.mjs`), 원격 URL 대상 성능 실측 스크립트(`client/scripts/perf-check.mjs`)가 이 시점에 추가됨. 실제 `flyctl deploy` 실행과 라이브 URL 발급은 사용자 계정으로 별도 진행. 이 과정에서 실제 커맨드 체계에 로봇 이동 목표를 배정하는 기능이 전혀 없어 라이브 서버에서 로봇이 걸어다니지 않는다는 갭도 발견돼, `sim_core`에 결정적 순찰 기능(`patrol_points`/`next_patrol_goal`, `server/src/sim.rs`)을 추가로 구현했다.
