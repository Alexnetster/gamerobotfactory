## 작업 방식

- **워크트리/브랜치 안 씀** — 솔로 저장소, 항상 `main`에서 바로 작업한다. 이건 사용자가 명시적으로 확정한 방침이다.
- **새 기능은 브레인스토밍 → 설계 스펙(`docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`) → 완결성 리뷰(독립 리뷰어가 갭/모순 점검) → 구현 계획(`docs/superpowers/plans/YYYY-MM-DD-<topic>-plan.md`, 태스크별 완전한 코드 포함) → subagent-driven-development로 태스크 하나씩 실행** 순서를 따른다.
- **각 태스크마다**: 구현자 서브에이전트 하나 → 스펙 컴플라이언스 리뷰어(계획서와 실제 코드를 라인 단위로 대조) → 코드 품질 리뷰어(뮤테이션 테스트 포함) → 지적사항 있으면 같은 구현자에게 수정 요청 → 재검증. 이 루프는 Plan 1~3(30개 태스크)과 로봇 내구도 기능(9개 태스크) 전체에서 실제로 여러 번 진짜 버그/공허한 테스트를 잡아냈다 — 생략하지 말 것.
- **테스트는 절대 self-report만 믿지 않는다.** 실제 컴파일된 바이너리에 진짜 WS/HTTP 클라이언트로 접속해서 검증(모킹 아님). "그럴듯해 보이지만 아무것도 검증 안 하는 테스트"가 최소 3번 실제로 나왔다(Plan 1 vacuous proptest, Plan 3 Lagged 통합테스트, 로봇 내구도 Task 1의 장애물 테스트) — 뮤테이션 테스트(동작을 일부러 깨고 테스트가 여전히 통과하는지 확인)로 걸러낸다.
- **태스크 완료 즉시 `docs/KANBAN.md` 갱신 + 커밋.** 미루면 파일이 실제 상태보다 낡아지고, 그 자체가 과거에 리뷰에서 지적당한 적 있다.
- 커밋은 `feat:`/`fix:`/`test:`/`docs:` 접두사, 항상 새 커밋(amend 안 함).
- 사용자는 보통 짧게("다음 진행해줘", "계속") 지시하고 판단은 맡긴다 — 매 단계 재확인받지 말고 계획대로 진행하되, 스코프를 벗어나거나 애매한 제품 결정(예: 어떤 동작이 의도된 것인지)은 리뷰에서 검증하고 필요하면 사용자에게 물어본다.
